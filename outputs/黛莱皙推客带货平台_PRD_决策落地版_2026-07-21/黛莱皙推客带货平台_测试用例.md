# 黛莱皙推客带货平台测试用例

## 一、文档说明

- 测试基线：[最新版 PRD v3.4](./黛莱皙推客带货平台_PRD_重写版_v3.4_0-1项目_本地评审稿.md)
- 测试对象：微信小程序、运营后台、微信官方能力衔接及关键服务端规则。
- 用例原则：一个用例只验证一个主要结果；金额、比例、限额和状态以后台配置或微信实际返回为准。
- 原型边界：手机号授权使用微信小程序原生组件，测试验证真实授权流程，不要求原型单独展示授权页面。
- 优先级：`P0` 为上线阻断项，`P1` 为核心功能，`P2` 为一般功能与体验。
- 执行结果：执行时填写“通过 / 失败 / 阻塞”，失败需关联缺陷编号。

## 二、测试角色与基础数据

| 类型 | 测试数据 |
| --- | --- |
| 小程序用户 | 游客、已登录未开通推广大使、已开通未签约、已签约、资格异常用户 |
| 后台账号 | 全菜单账号、内容管理菜单账号、财务管理菜单账号、无对应菜单账号、已封禁账号 |

| 订单 | 待结算、已结算、已失效、自购、退款、归因不足订单 |
| 提现 | 待审核、审核通过、驳回、处理中、到账、退回记录 |

## 六、上线验收门槛

- 所有 `P0` 用例必须通过，且无未关闭的阻断或严重缺陷。
- `P1` 用例通过率不低于项目约定标准；失败项必须有明确修复版本或业务接受结论。
- 微信官方开通、分享、视频号跳转、订单归因和提现支付能力必须使用真实环境完成至少一轮端到端验证。
- 佣金、好友分佣、退款、提现、税费和余额变动需完成业务明细与财务结果对账。
- 上线前完成一级菜单权限矩阵、敏感信息脱敏、导出访问控制、操作日志和账号注销专项验收。
